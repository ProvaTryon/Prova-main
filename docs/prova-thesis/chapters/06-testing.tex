\section{Testing Strategy}
\label{sec:test-strategy}

The testing strategy for \projectname\ follows a layered approach that reflects the system's service-oriented architecture. Testing is organized into four levels:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Unit Testing.} Individual functions, validators, and service methods are tested in isolation using mock objects to replace external dependencies. This level verifies correctness of business logic without requiring running services.

    \item \textbf{Integration Testing.} End-to-end request flows are tested using FastAPI's \texttt{TestClient} and mock service backends. These tests verify that routers, middleware, validators, and service clients interact correctly, including error propagation and response formatting.

    \item \textbf{Performance Testing.} Response times and throughput are measured for AI-intensive pipelines under realistic workloads to ensure they meet the non-functional requirements defined in Chapter~3.

    \item \textbf{Usability Evaluation.} User feedback sessions evaluate the perceived quality of try-on results, measurement accuracy, and chatbot helpfulness.
\end{enumerate}

The try-on service uses \textbf{pytest} as its testing framework, configured with \texttt{pytest-cov} for code coverage measurement. The coverage threshold is set to 70\% as defined in \texttt{pytest.ini}. Test discovery follows the convention of \texttt{test\_*.py} files in the \texttt{tests/} directory.

\section{Unit Testing}
\label{sec:test-unit}

\subsection{Test Infrastructure}

The try-on service's test suite employs a dependency injection pattern to isolate tests from external services. The \texttt{conftest.py} module (274 lines) defines four mock client classes that replace the real service clients during testing:

\begin{itemize}[leftmargin=*]
    \item \textbf{MockColabClient} --- Returns fixed byte data for virtual try-on requests, simulating Prova-VTON inference without GPU access.
    \item \textbf{MockMultiviewClient} --- Simulates the multiview pipeline responses including upload, batch processing, and data listing operations.
    \item \textbf{MockVideoPreprocessor} --- Returns pre-defined preprocessing results without invoking YOLOv8, DensePose, or human parsing models.
    \item \textbf{MockTrueFitClient} --- Simulates multi-angle try-on with configurable failure scenarios. Supports partial view failures for testing graceful degradation.
\end{itemize}

These mocks are injected into the FastAPI application through dependency overrides. The \texttt{client} fixture creates a \texttt{TestClient} instance with all four overrides applied and clears them after each test:

\begin{verbatim}
app.dependency_overrides[get_colab_client] = lambda: mock_colab_client
app.dependency_overrides[get_multiview_client] = lambda: mock_multiview_client
app.dependency_overrides[get_preprocessing_service] = lambda: mock_preprocessor
app.dependency_overrides[get_truefit_client] = lambda: mock_truefit_client
\end{verbatim}

Test data is provided through fixtures: \texttt{sample\_image\_bytes} generates a 512$\times$512 red JPEG image using Pillow, and \texttt{sample\_video\_bytes} produces synthetic video data.

\subsection{Validator Testing}

Unit tests verify the input validation layer, ensuring that:
\begin{itemize}[leftmargin=*]
    \item Image file extensions are restricted to supported formats (JPEG, PNG, WebP).
    \item File size limits are enforced (configurable per endpoint).
    \item Height values are constrained to the 50--300~cm range.
    \item Weight values are constrained to the 20--500~kg range.
    \item Gender values are normalized to the expected set (male, female, unisex).
    \item Person and cloth identifiers conform to the expected naming pattern.
\end{itemize}

\subsection{Exception Handling Testing}

The exception handling module is tested to verify that custom exception types (\texttt{ServiceUnavailableError}, \texttt{ExternalAPIError}) are correctly raised and mapped to appropriate HTTP status codes (503 and 502, respectively) by the global exception handler middleware.

\subsection{TrueFit Pipeline Testing}

The TrueFit test suite (\texttt{test\_truefit.py}, 130 lines) validates the multi-angle try-on pipeline through four integration tests:

\begin{enumerate}[leftmargin=*]
    \item \textbf{Default Contract Test.} Submits a person image and garment image with no optional parameters. Verifies that the response has HTTP 200 status, \texttt{image/png} content type, and contains the expected image bytes.

    \item \textbf{Control Parameters Test.} Submits a request with explicit control parameters (\texttt{garment\_type=dress}, \texttt{garment\_source\_mode=model\_worn}, \texttt{shirt\_placement=untucked}, \texttt{view\_hint=front}). Verifies that all parameters are correctly propagated to the underlying client by inspecting the mock's recorded keyword arguments.

    \item \textbf{Multi-Angle Partial Success Test.} Configures the mock client to fail on the ``right'' view while succeeding on the other three views. Submits a multi-angle request with ZIP response format. Verifies that the ZIP archive contains \texttt{front.png}, \texttt{left.png}, \texttt{back.png} (but not \texttt{right.png}) and a \texttt{metadata.json} file with status \texttt{partial\_success} and \texttt{failed\_views: ["right"]}.

    \item \textbf{Total Failure Test.} Configures all four views to fail. Verifies that the endpoint returns HTTP 502 with a structured error response listing all four failed views.
\end{enumerate}

\section{Performance Testing}
\label{sec:test-performance}

Performance testing evaluates the latency and throughput of the AI subsystems under typical conditions. Metrics are collected through analytics tracking injected at the service endpoints.

\subsection{Try-On Pipeline Latency}

The virtual try-on pipelines exhibit significant latency differences due to their underlying architectures and deployment configurations:

\begin{itemize}[leftmargin=*]
    \item \textbf{2D Try-On (Prova-VTON).} Running on a remote Colab server (T4 GPU), single-image inference averages 12--18 seconds per request, including image upload and download overhead.
    \item \textbf{Multi-Angle Try-On (TrueFit).} Using OpenRouter's API, the two-stage process averages 8--12 seconds per view. The multi-angle endpoint processes up to two views concurrently (via the semaphore limit), resulting in a total time of 20--30 seconds for all four viewpoints (front, right, left, back). Garment extraction caching saves approximately 4 seconds per view compared to naive sequential processing.
    \item \textbf{Multiview Pipeline (Prova-VTON).} The Prova-VTON pipeline involves heavy preprocessing and remote inference. Frame extraction and local preprocessing (YOLOv8, DensePose, SCHP) take 40--60 seconds depending on video length. Remote inference and dataset transfer add an additional 60--90 seconds. The complete end-to-end process typically requires 2--3 minutes.
\end{itemize}

\subsection{RAG Pipeline Latency}

The RAG fashion assistant pipeline is optimized for interactive conversational speeds. Timing logs from the orchestrator provide the following latency breakdown for a typical product search query:

\begin{itemize}[leftmargin=*]
    \item \textbf{Classification.} Fast LLM call: $\sim$0.5--1.0 seconds.
    \item \textbf{Intent Extraction.} Structured LLM call: $\sim$1.5--2.5 seconds.
    \item \textbf{Retrieval.} Metadata filtering + semantic ranking: $\sim$0.1--0.3 seconds.
    \item \textbf{Outfit Assembly.} In-memory scoring and slot assignment: $<0.1$ seconds.
    \item \textbf{Response Generation.} Natural language generation: $\sim$2.0--3.0 seconds.
\end{itemize}

The total conversational latency averages 4.0--7.0 seconds, which is acceptable for a complex retrieval task and within the 10-second non-functional requirement. Non-fashion queries (greetings, clarification) bypass the retrieval stages and respond within 1.0--2.0 seconds.

\subsection{Body Measurement Performance}

The body measurement pipeline using SHAPY demonstrates consistent latency. When running locally on an NVIDIA RTX 3050 GPU, the end-to-end request (including image validation, mesh regression, and circumference derivation) completes in 8--12 seconds. CPU-only inference falls back to approximately 45--60 seconds, validating the requirement for hardware acceleration in production deployments.

\section{Usability Evaluation}
\label{sec:test-usability}

Usability evaluation focused on user interaction with the core AI features. A small group of test users interacted with the frontend application and provided feedback.

\subsection{Interface and Localization}

The bilingual interface successfully adapts to the user's preferred language. The automatic RTL layout mirroring via Tailwind CSS was verified to correctly flip directional elements without breaking responsive grid layouts. Users noted that the ability to switch languages without losing session state or cart contents was a significant benefit.

\subsection{Virtual Try-On Experience}

Evaluation of the try-on features highlighted the trade-offs between the different pipelines:
\begin{itemize}[leftmargin=*]
    \item The \textbf{Prova-VTON} pipeline was praised for its photorealism and texture preservation, particularly for complex garment patterns. However, it sometimes struggled with unusual poses or heavy occlusions.
    \item The \textbf{TrueFit Multi-Angle} pipeline provided a more comprehensive understanding of garment fit from different sides, which users found highly valuable. The generative approach sometimes resulted in minor hallucinations (e.g., altering background details or accessories), but the garment representation remained consistent.
\end{itemize}

\subsection{RAG Assistant Interaction}

Users interacted with the fashion assistant using natural language queries involving complex constraints (e.g., ``I need a blue dress for a summer wedding under \$100''). The system successfully extracted multiple intent attributes and returned relevant outfits. The bilingual response generation correctly matched the input language, although users occasionally noted that highly specific fashion terminology was sometimes translated literally into Arabic.

\section{Quantitative Evaluation}
\label{sec:test-quantitative}

To validate the generative quality and view-consistency of the Prova-VTON multiview pipeline, a quantitative evaluation was performed against several state-of-the-art 3D and multiview generation baselines (DreamWaltz, TIP-Editor, and GaussCtrl) using the \textbf{MVHumanNet} dataset. 

The evaluation measures two primary aspects:
\begin{itemize}[leftmargin=*]
    \item \textbf{Texture Preservation:} Measured using CLIP consistency (CLIPcons) and DINO similarity (DINOsim) to evaluate how well the generated garments retain the visual identity and texture of the original clothing.
    \item \textbf{Geometry Alignment:} Evaluated via human preference voting (Votequality and Votealign) to determine structural correctness and multiview coherence.
\end{itemize}

As shown in Table~\ref{tab:quantitative-evaluation}, while the baseline models struggle significantly with multiview geometry alignment (scoring below 3\% in human preference), \projectname\ achieves state-of-the-art structural consistency. Furthermore, it achieves the highest DINOsim score (0.623) while maintaining highly competitive CLIPcons, demonstrating the effectiveness of the multiview conditioning approach.

\section{Summary of Results}
\label{sec:test-results}

The testing and evaluation phase confirmed that \projectname\ successfully implements its core functional requirements:

\begin{itemize}[leftmargin=*]
    \item The \textbf{backend} handles concurrent requests, secure authentication, and data persistence reliably.
    \item The \textbf{frontend} provides a responsive, bilingual interface with robust state management.
    \item The \textbf{virtual try-on} service successfully integrates multiple AI pipelines, offering trade-offs between speed, photorealism, and multi-view capability. The weight-based correction mechanism in the measurement pipeline significantly improved circumference accuracy compared to raw SHAPY predictions.
    \item The \textbf{RAG assistant} correctly interprets complex multi-constraint queries, performs hybrid retrieval, and assembles coherent outfit recommendations with contextual explanations.
\end{itemize}

Integration tests confirm that the microservice architecture handles service failures gracefully, returning structured errors rather than crashing the orchestrator. Performance metrics indicate that while the AI pipelines are compute-intensive, the system architecture manages the latency effectively through asynchronous processing and informative loading states.
